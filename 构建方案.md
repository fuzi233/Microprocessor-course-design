# 自主导航小车 — 分阶段构建方案

## 项目概况

- **平台**: STM32F446RETX (Cortex-M4, 180MHz)
- **传感器**: MPU6500 (IMU) + RPLidar (激光雷达)
- **执行器**: 直流减速电机 × 2 + 编码器
- **RTOS**: FreeRTOS
- **前提**: 底层硬件驱动（motor_driver, mpu6500, rplidar, oled_ssd1306, button）已验证正常

## 最终交付物要求

项目必须包含以下核心能力：

| 必需内容 | 说明 |
|----------|------|
| **FreeRTOS** | 实时多任务调度，队列/信号量/互斥锁 |
| **DMA** | 雷达数据高速接收 (UART DMA) + IMU 数据读取 (I2C DMA) |
| **Mapping** | 占据栅格地图构建 + 障碍膨胀 |
| **A\*** | 全局路径规划 |

---

## 模块依赖关系

```
DMA传输 ──→ Mapping ──→ A*规划 ──→ Path Tracker ──→ Frontier探索
    │            │                                      │
    └──→ FreeRTOS（贯穿所有层）  ←─────────────────────────┘
                                ←──→ Local Avoidance（横切）
```

关键逻辑链：

> 没有 DMA 就无法高效接收雷达数据 → 没有雷达数据就无法建图 → 没有地图就无从规划路径 → 没有路径就谈不上跟踪和探索

---

## 分阶段构建计划

### 阶段 1：FreeRTOS 基础框架

**优先级**: 最高 — 所有后续模块的调度地基

**要写的内容**:

- 任务优先级体系设计（IMU > 雷达 > 控制 > 显示）
- Queue 用于任务间数据传递（雷达帧队列、控制命令队列）
- Mutex 保护 I2C 总线共享（OLED 和 MPU6500 共用 I2C1）
- Semaphore 用于事件同步（数据就绪通知）
- 可选：Software Timer 用于周期性任务

**测试方法**:

- 创建 2-3 个简单任务（LED 闪烁 + UART 周期打印）
- 验证 `vTaskDelayUntil` 精确周期调度（示波器看 GPIO 翻转）
- 验证 Queue 发送/接收不丢数据（发送 1000 次，接收计数校验）
- 验证 Mutex 在两个任务争抢 I2C 时不发生死锁

**验收标准**: 调度器稳定运行 30 分钟，任务间通信无死锁、无数据丢失

---

### 阶段 2：DMA 传输层

**优先级**: 高 — Mapping 的前置依赖

**要写的内容**:

- USART3 DMA 循环接收（激光雷达 460800 bps 高速数据流）
- I2C1 DMA 读取 MPU6500 陀螺仪数据（确认现有实现无误）
- ADC DMA 读取电位器（确认现有实现无误）
- DMA 半满中断 + 全满中断回调 → 数据写入 Ring Buffer
- Ring Buffer 无锁实现（Write Index 在 ISR 更新，Read Index 在 Task 更新）

**测试方法**:

- 通过 UART2 将 DMA 缓冲区原始字节打印到 PC
- 确认连续接收 10 万字节不丢包
- 发送 `RPLIDAR_CMD_SCAN` 后，观察响应报文是否完整（起始字节 0xA5 0x5A）
- 用逻辑分析仪抓 USART3 RX 引脚，对比 DMA 收到的数据

**验收标准**: 雷达全速扫描下 DMA 不掉帧，环形缓冲不溢出

---

### 阶段 3：Mapping（栅格地图构建）

**优先级**: 高 — A* 的直接输入，是整个导航链的核心数据结构

**依赖**: 阶段 1（FreeRTOS）、阶段 2（DMA）

**要写的内容**:

- `grid_map.c` / `grid_map.h`
- 占据栅格地图（建议 200×200 格，分辨率 50mm/格 → 覆盖 10m×10m）
- 三种栅格状态: `UNKNOWN` / `FREE` / `OCCUPIED`
- 射线投射（Ray Casting）更新逻辑:
  - 雷达位置到击中点之间的所有栅格 → FREE
  - 击中点栅格 → OCCUPIED
  - 超出量程的射线 → 整条射线 FREE
- 障碍膨胀（Inflate）: 以安全半径对障碍栅格做像素级膨胀
- 坐标转换: `WorldToCell()` / `CellToWorld()`

**测试方法**:

1. **PC 端模拟测试（优先）**:
   - 使用 `Algo/Host/nav_validation_main.c`
   - 喂入模拟激光数据（一圈 360 点，已知障碍布局）
   - 验证建图结果与预期布局一致
2. **上板静态测试**:
   - 小车放在固定位置，周围放置纸箱
   - 转动雷达 360°，通过 UART 打印地图统计:
     - 已知空闲格数量
     - 障碍格数量
     - 未知格数量
3. **可视化验证**:
   - 通过串口将地图矩阵发回 PC
   - Python 脚本接收后用 matplotlib 画热力图
   - 人工比对障碍物位置是否与实际环境一致

**验收标准**: 静态环境下建图精度 ±1 格（50mm），无 ghost obstacle（虚假障碍）

---

### 阶段 4：A* 全局路径规划

**优先级**: 中高 — 导航"大脑"，决定小车走向

**依赖**: 阶段 3（Mapping 结果正确）

**要写的内容**:

- `planner_astar.c` / `planner_astar.h`
- 8 邻域搜索（允许斜向移动，cost = √2 ≈ 1.414）
- 对角距离启发函数（Diagonal Distance Heuristic）
- Open List 用二叉堆实现（嵌入式内存受限，不宜用 STL）
- `CompressPath()`: 共线节点合并，减少路径点数
- 边界处理: 起点/终点在障碍上 → 搜索最近空闲格替代
- 超时保护: 搜索超过 N 次迭代强制终止

**测试方法**:

1. **PC 端先测**:
   - 在 `nav_validation_main.c` 中构造已知地图
   - 测试 20+ 组起终点组合
   - 验证路径不与障碍碰撞
2. **上板测**:
   - 先手动建图，然后给定起终点坐标
   - 通过 UART 打印完整路径节点序列
3. **边界 case**:
   - 起点 = 终点 → 应直接返回成功（路径长度 1）
   - 终点不可达 → 应返回失败，不死循环
   - 迷宫式环境 → 应找到通路

**验收标准**: 任意可达起终点在 100ms 内返回有效无碰撞路径

---

### 阶段 5：Path Tracker（路径跟踪）

**优先级**: 中 — 将规划结果转化为运动控制

**依赖**: 阶段 4（A* 输出路径）

**要写的内容**:

- `path_tracker.c` / `path_tracker.h`
- Pure Pursuit 风格前视点跟踪
- 前视距离自适应（弯道缩近、直道放远）
- 线速度/角速度计算公式
- 转弯降速策略（曲率越大速度越低）
- 终点减速策略（接近目标时逐渐降速）

**测试方法**:

- 车轮架空，发送简单直线路径 → 验证左右轮目标速度对称
- 发送 90° 弯道路径 → 验证差速转向指令方向正确
- 发送已到达的目标点 → 验证停车指令输出
- 编码器反馈速度与目标速度对比，确认 PID 环路正常

**验收标准**: 直线段误差 < 50mm，到达目标点自动停车

---

### 阶段 6：Frontier 探索 + 导航运行时

**优先级**: 中 — 实现全自主探索

**依赖**: 阶段 3、4、5 全部验证通过

**要写的内容**:

- `frontier.c` / `frontier.h`
  - 扫描全图找到 FREE 与 UNKNOWN 的边界栅格
  - BFS 聚类相邻 Frontier 栅格
  - 聚类打分: 大小（信息收益）vs 距离（代价）
  - 选择最优探索目标
- `nav_runtime.c` / `nav_runtime.h`
  - 状态机: `IDLE → WAIT_SCAN → MAPPING → PLANNING → TRACKING → EXPLORING → FINISHED`
  - 触发条件: 目标到达 / 路径被阻断 / 超时重规划
  - 衔接所有模块: Map → Frontier → A* → Tracker → Motor

**测试方法**:

- 小车放入半未知环境（已知 30%，未知 70%）
- 观察是否能:
  1. 识别已探索/未探索边界
  2. 生成合理的探索目标
  3. 规划并执行到目标
  4. 更新地图后重新选择新目标
- OLED / UART 实时显示状态机当前状态

**验收标准**: 在 5m×5m 范围内 3 分钟内完成 90% 以上区域探索

---

### 阶段 7：Local Avoidance（局部避障）+ 综合联调

**优先级**: 中低 — 安全性保障

**依赖**: 全部前置阶段

**要写的内容**:

- `local_avoidance.c` / `local_avoidance.h`
- 扇区法: 将激光 360° 划分为 12 个扇区（每扇区 30°）
- 三个距离环:
  - 紧急区（<200mm）→ 立即停车
  - 减速区（200mm-350mm）→ 降速 + 转向
  - 安全区（>350mm）→ 正常行驶
- 扇区打分: 选择最安全的方向偏转
- 不修改全局路径，只在命令层做速度修正

**测试方法**:

1. 静态环境全自主导航测试
2. 路径中突现障碍（人走过去）→ 验证避障响应
3. 低速实地上路联调
4. 5 次完整探索任务，记录成功率

**验收标准**: 动态障碍出现 500ms 内做出响应，不碰撞

## 测试原则

1. **PC 优先**: 所有算法逻辑先在 `Algo/Host/nav_validation_main.c` 上验证通过，再上板
2. **单步验证**: 每个阶段确认无误后再进入下一个阶段
3. **可视化驱动**: 地图和路径数据通过串口发回 PC 可视化，不盲调
4. **架空 → 落地**: 先架空轮测试电机指令，确认安全后再落地

## 风险提示

- **I2C 总线冲突**: OLED 和 MPU6500 共用 I2C1，Mutex 保护是必须的
- **DMA 与 Cache 一致性**: STM32F4 无 D-Cache，但需注意 DMA 缓冲区对齐
- **堆栈溢出**: 导航算法栈消耗大，`configMINIMAL_STACK_SIZE` 可能需要上调
- **雷达盲区**: RPLidar 最近检测距离约 150mm，车身近处存在盲区
